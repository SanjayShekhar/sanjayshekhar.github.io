    Why Code Mode Beats Raw MCP and Open Interpreter

    There are three popular ways to let a language model actually do something instead of just talking about it. You can expose tools to it directly over MCP and let it call them. You can hand it a code interpreter and let it run whatever it writes, the Open Interpreter approach. Or you can do something in between: turn your tools into a code API and let the model write code that calls them. That last option, which Cloudflare has branded "Code Mode," is the one I think wins for most serious agent work, and the reasons are not just architectural taste. There is benchmark evidence behind it.

    Let me lay out why.

    The three approaches in one paragraph each

    Direct MCP tool calling. The Model Context Protocol gives agents a uniform way to discover and invoke external tools. In the standard setup, each tool is presented to the model directly, and the model emits a special tool-call token followed by a JSON blob naming the tool and its arguments. The harness parses that, runs the tool, and feeds the result back into the model. It is clean, standardized, and works. The catch is that those tool-call tokens are not something models saw much of during pretraining. They are trained on them after the fact, usually with synthetic data, so models are noticeably weaker at producing them than at producing ordinary text or code.

    Open Interpreter. This open-source project gives the model a pair of hands on your actual machine. You describe a task in plain English, the model writes Python, JavaScript, or shell code, and Open Interpreter runs it locally with your approval, with full access to your files and environment. It is fantastic for interactive exploration and one-off automation. But it works by handing the model your whole computer, which is exactly where its weaknesses come from: it runs with broad local access, it is non-deterministic, and in its default online mode it ships data to its own hosted services for code hints and screen reading.

    Code Mode. Here you take the same MCP tools, convert their schemas into a typed code API (TypeScript, in Cloudflare's implementation), and ask the model to write code against that API. The code runs in an isolated sandbox whose only window to the outside world is the bindings representing your connected MCP servers. The model gets to play to its strongest skill, writing code, while you keep the tight, standardized tool surface that MCP gives you.

    The core insight: models write code better than they emit tool calls

    This is the load-bearing claim, so it deserves evidence rather than vibes.

    The cleanest study is the CodeAct paper, "Executable Code Actions Elicit Better LLM Agents" (ICML 2024). The authors compared three ways of expressing an agent's actions: text, JSON function calls, and executable Python code. Across 17 different open and proprietary models, code actions came out ahead. On simple one-tool tasks the formats were roughly comparable, but on their multi-tool, multi-turn benchmark (M3ToolEval), code actions delivered up to about 20 percentage points higher success than the JSON and text baselines. Just as important for cost, code actions needed roughly 30 percent fewer steps to finish a task, because the model could compose several operations in a single block instead of round-tripping through the harness once per call.

    The reasons are intuitive once you see them. Code gives the model control flow (loops, conditionals) and data flow (variables, intermediate results) inside a single action. It lets the model reuse functions from libraries it already knows. And critically, pretraining is saturated with real code from millions of projects, while tool-call formats are a thin, mostly synthetic slice of training data. As the paper puts it, optimizing for code actions is a better route to tool use than fine-tuning toward JSON, because the models already arrive with strong code ability baked in.

    Cloudflare reached the same conclusion from production experience rather than a benchmark, and framed it memorably: making a model do tasks through tool-calling is like teaching Shakespeare a month of Mandarin and then asking for his best play. The talent is there. You are just forcing it through a channel it was never fluent in.

    It is worth being honest about the ceiling here too. Even on the dedicated tool-calling leaderboards, the top models top out around 75 percent overall accuracy on the current Berkeley Function Calling Leaderboard (BFCL v4), which means direct tool calling still fails a meaningful fraction of the time. That headroom is exactly what a better action format is trying to recover.

    Where Code Mode beats raw MCP

    If models are better at code, then wrapping MCP tools in a code API and letting the model write against it captures that advantage directly. Three concrete wins follow.

    It scales to many tools. Present a model with dozens of raw tools and it starts to struggle to pick the right one. Present the same capabilities as a typed API with doc comments and it behaves much more like a coding assistant browsing a library, which it is good at.

    It stops wasting tokens on hand-offs. With direct calling, every intermediate result has to pass back through the model just to be copied into the next call. With code, the model writes a script that chains the calls, keeps the intermediate data in variables, and only reads back the final answer. That is the same 30-percent-fewer-steps effect CodeAct measured, and since model calls usually dominate agent cost, it is a real bill reduction, not just elegance.

    It keeps MCP's best property. You do not lose anything that made MCP good. The uniform discovery, the standardized auth handled out of band, the self-describing schemas: all of that still drives the generated code API. You are changing how the model consumes the tools, not throwing away the protocol.

    Where Code Mode beats Open Interpreter

    Open Interpreter already embraces the "let the model write code" insight, so the gap here is not about action format. It is about safety and control.

    Sandboxing by default, not by trust. Open Interpreter's power is that it runs code on your real machine with access to your files, which is also its danger. Code Mode inverts this. The sandbox starts with no general internet or filesystem access at all, and the only way out is through the specific MCP bindings you granted. You do not have to reason about what the model might touch, because the boundary is defined by the API surface you exposed.

    No keys to leak. Because the sandbox reaches tools through bindings rather than by making authenticated network calls itself, the access tokens live with the supervisor and never enter the model's reach. The model literally cannot write code that exfiltrates a key it never sees. With a local interpreter holding your environment variables, that guarantee is much harder to make.

    Built for agents, not just a human at a terminal. Open Interpreter shines as an interactive co-pilot where a person approves each step and the non-determinism is fine. Code Mode is designed for unattended agent loops: disposable, fast-starting sandboxes (Cloudflare uses lightweight isolates that spin up in milliseconds), a constrained tool surface, and no human in the approval path on every action.

    Side by side

    Direct MCP Open Interpreter Code Mode
    Action format JSON tool calls Code (local) Code against a tool API
    Plays to model's code strength No Yes Yes
    Handles many/complex tools well Weakens fast N/A (open-ended) Yes
    Multi-step efficiency Round-trips per call Good Good (chains in one block)
    Default isolation Harness-dependent Full local access Sandboxed, bindings only
    Secrets exposure Possible Possible Hidden behind bindings
    Best fit Simple, few-tool agents Interactive human-in-loop Unattended multi-tool agents

    The honest caveats

    This is an argument, not a sermon, so here is the other side.

    Direct MCP is perfectly fine when you have a handful of tools and short chains. The extra machinery of a sandbox and a generated API is not worth it for a weather lookup. For frontier models specifically, the raw tool-calling gap has narrowed since these ideas first appeared, so the advantage is largest for smaller and open-weight models and for genuinely complex, many-step tasks.

    Open Interpreter remains the better choice when the whole point is local access: cleaning a messy folder on your own disk, prototyping a data analysis interactively, or driving software on your actual machine. Code Mode's isolation is a feature for agents and a limitation if local reach is the goal.

    And Code Mode is not free. You need a real sandbox, a schema-to-code conversion step, and a runtime that can load and discard code quickly. Cloudflare can lean on its isolate platform for that; not every stack has an equivalent, though local sandboxes and ephemeral containers can stand in.

    Bottom line

    The single most useful fact in this whole debate is that language models are simply better at writing code than at emitting tool calls, and that is measurable, not aesthetic: up to 20 percent higher success and around 30 percent fewer steps in the CodeAct experiments. Direct MCP ignores that fact. Open Interpreter embraces it but pays for local power with weak isolation. Code Mode keeps the code-writing advantage and the standardized MCP tool surface while adding sandboxing and secret-hiding on top. For unattended agents that juggle many tools across many steps, that combination is hard to beat.
